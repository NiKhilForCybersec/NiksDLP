Symantec DLP Troubleshooting

This guide covers common issues with Symantec DLP including endpoint agent problems, Enforce Server issues, Oracle database connectivity, and detection failures.

Endpoint Agent Issues

Symptoms: Agent appears offline or last seen timestamp is stale.

Diagnostic Steps:

Command Line 
# Check agent status
edpactl status

# Test connectivity to Enforce Server
edpactl test-connection

# Check agent logs for errors
# Windows:
type "C:\ProgramData\Symantec\DataLossPrevention\Agent\logs\communication.log"
# Linux/macOS:
cat /var/log/Symantec/DataLossPrevention/Agent/communication.log

Common Causes & Solutions:

  • Firewall blocking port 8200 — Verify port is open: telnet enforce-server 8200
  • DNS resolution failure — Check Enforce Server hostname resolves correctly
  • Certificate issues — Verify SSL certificate chain is valid
  • Agent service stopped — Restart service: edpactl restart

Diagnostic Steps:

  1. Check which process is consuming CPU (edpa.exe, vontu*.exe)
  2. Review recent activity—large file scan or many small files?
  3. Check for antivirus conflicts

Solutions:

  • Add file/folder exclusions — Exclude temp folders, build directories
  • Exclude processes — Add high-IO processes to exclusion list
  • Adjust scan settings — Reduce real-time scanning scope
  • Schedule intensive scans — Run Endpoint Discover during off-hours
Agent Exclusions (channels.conf) 
# Add to channels.conf
ExcludedProcesses=devenv.exe,java.exe,sqlservr.exe
ExcludedFolders=C:\Windows\Temp,C:\Users\*\AppData\Local\Temp
ExcludedFileTypes=*.tmp,*.log,*.bak

Verification Steps:

Command Line 
# Check policy version on agent
edpactl show-config | grep PolicyVersion

# Force policy update
edpactl update-policies

# Verify agent group assignment
edpactl show-config | grep AgentGroup

Common Causes:

  • Agent in wrong agent group
  • Policy not deployed to agent group
  • Network issues preventing policy download
  • Policy file corrupted—force re-download

Symptoms: Web uploads to HTTPS sites not being detected.

Diagnostic Steps:

  1. Verify HTTPS inspection is enabled in agent config
  2. Check that DLP root CA is in trusted store
  3. Test with specific browser (Edge/Chrome work best)
  4. Check for proxy conflicts
PowerShell - Check Certificate 
# Check for DLP certificate in trusted root
Get-ChildItem Cert:\LocalMachine\Root | 
    Where-Object { $_.Subject -like "*Symantec*" -or $_.Subject -like "*DLP*" }

# Import certificate if missing
Import-Certificate -FilePath "DLP_Root_CA.cer" -CertStoreLocation Cert:\LocalMachine\Root

Enforce Server Issues

Diagnostic Steps:

  1. Check Enforce Server CPU and memory usage
  2. Review database performance
  3. Check disk I/O on Enforce Server
  4. Review concurrent user count
Linux - Performance Check 
# Check DLP services
systemctl status SymantecDLP*

# Check Java heap usage
jstat -gc $(pgrep -f "ProtectManager") 1000 5

# Review Enforce logs
tail -f /var/log/Symantec/DataLossPrevention/Enforce/tomcat/logs/catalina.out

# Check disk space
df -h /var/Symantec

Common Solutions:

  • Increase JVM heap size in startup scripts
  • Archive old incidents to reduce database size
  • Add indexes to frequently queried columns
  • Schedule reports during off-hours

Service Startup Order:

  1. Oracle Database (must be running first)
  2. Symantec DLP Manager Service
  3. Symantec DLP Notifier Service
  4. Detection Servers (after Manager)
Service Management 
# Check all DLP services
systemctl list-units --type=service | grep -i symantec

# Start services in order
systemctl start oracle-xe
systemctl start SymantecDLPManager
systemctl start SymantecDLPNotifier

# Check for port conflicts
netstat -tlnp | grep -E "443|8443|8200"

Oracle Database Issues

Common Error Messages:

  • ORA-12541: TNS:no listener — Listener not running
  • ORA-12514: TNS:listener does not know of service — Wrong service name
  • ORA-01017: invalid username/password — Credentials issue
  • ORA-00257: archiver error — Archive log full
Oracle Diagnostics 
# Check listener status
lsnrctl status

# Start listener if down
lsnrctl start

# Test database connection
sqlplus protect/password@DLPDB

# Check database status
sqlplus / as sysdba
SQL> SELECT STATUS FROM V$INSTANCE;
SQL - Check and Extend Tablespace 
-- Check tablespace usage
SELECT 
    tablespace_name,
    ROUND(used_space * 8192 / 1024 / 1024 / 1024, 2) AS used_gb,
    ROUND(tablespace_size * 8192 / 1024 / 1024 / 1024, 2) AS total_gb,
    ROUND(used_percent, 2) AS pct_used
FROM dba_tablespace_usage_metrics
ORDER BY used_percent DESC;

-- Add datafile to extend tablespace
ALTER TABLESPACE PROTECT_DATA 
ADD DATAFILE '/u01/app/oracle/oradata/DLPDB/protect_data02.dbf' 
SIZE 10G AUTOEXTEND ON MAXSIZE 50G;

-- Check largest tables (for cleanup planning)
SELECT 
    owner, 
    segment_name, 
    ROUND(bytes/1024/1024/1024, 2) AS size_gb
FROM dba_segments 
WHERE owner = 'PROTECT'
ORDER BY bytes DESC
FETCH FIRST 10 ROWS ONLY;
SQL - Performance Diagnostics 
-- Find slow queries
SELECT 
    sql_text,
    elapsed_time/1000000 AS elapsed_seconds,
    executions,
    elapsed_time/executions/1000 AS avg_ms
FROM v$sql
WHERE parsing_schema_name = 'PROTECT'
AND executions > 0
ORDER BY elapsed_time DESC
FETCH FIRST 20 ROWS ONLY;

-- Check for missing indexes
SELECT 
    table_name,
    index_name,
    status
FROM dba_indexes
WHERE owner = 'PROTECT'
AND status != 'VALID';

-- Rebuild invalid indexes
ALTER INDEX PROTECT.IDX_INCIDENT_DATE REBUILD;

Detection Issues

Troubleshooting Steps:

  1. Test policy using the Policy Tester in Enforce Console
  2. Verify file type is supported for content inspection
  3. Check that channel is enabled and monitored
  4. Review rule conditions and thresholds
  5. Check for conflicting exceptions

Common Issues:

  • Encrypted files — Cannot inspect encrypted/password-protected files
  • Unsupported format — Some file types not inspected
  • Threshold too high — Reduce minimum match count
  • Regex too restrictive — Test pattern against sample data

IDM Index Issues:

  • Verify index is up to date (check last indexed timestamp)
  • Ensure source documents haven't changed significantly
  • Check match percentage threshold (40% default may be too high)
  • Re-index if source documents were modified

EDM Index Issues:

  • Verify source data export completed successfully
  • Check column mappings are correct
  • Ensure data format matches expected (dates, numbers)
  • Verify multi-column matching requirements
SQL - Check Index Status 
-- Check EDM profile status
SELECT 
    profile_name,
    record_count,
    last_indexed,
    status
FROM protect.edm_profiles;

-- Check IDM profile status
SELECT 
    profile_name,
    document_count,
    last_indexed,
    status
FROM protect.idm_profiles;
